[COLOR yellow][B]v0.1.22[/B][/COLOR]
* Fix NOX skin being auto-disabled as "incompatible add-on" right after
  switching to it (Kodi then fell back to plain Estuary, so it looked
  identical to the simple skin). Cause: skin.povil.nox declared
  <import addon="xbmc.gui" version="5.15.0"/> (Nexus), which Kodi 21
  Omega rejects. The build's working skins (Estuary, FENtastic) keep that
  import commented out; NOX now matches. Skin bumped to 1.0.1 and
  NOX_SKIN_VERSION gated to 1.0.1 so devices that already pulled the
  broken 1.0.0 re-download the fixed pack on the next switch.

[COLOR yellow][B]v0.1.21[/B][/COLOR]
* Fix the "Kodi hangs for a few seconds on the first launch after an
  install / quick update, force-stop required" bug. startup.py is an
  xbmc.service with start="startup", so it runs before Kodi's GUI
  exists; the first-launch modal dialogs (build notification, skin-
  switch notification) were doModal()'d that early and deadlocked the
  boot. Because those blocks set their one-shot dismiss flag BEFORE
  showing, the hang happened exactly once and a force-stop recovered --
  matching the symptom. Added wait_for_gui_ready(): the service now
  blocks (bounded) until the Home window is actually live before any
  first-launch dialog, so each modal has a real parent window.

[COLOR yellow][B]v0.1.20[/B][/COLOR]
* New 4th skin: NOX (skin.povil.nox), a rebranded + scrubbed Estuary MOD
  with a Hebrew home menu remapped to our POV / idanplus / otaku addons.
  Like Arctic Fuse 3 it is NOT bundled in the base build -- it downloads
  on demand the first time the user picks it in Switch Skin, so it costs
  nothing for users who do not use it. The on-demand installer was
  generalised (_ensure_packs_installed) so AF3 and NOX share one path,
  including the critical Addons-DB register step that prevents the silent
  Estuary fallback. The side-loader and all foreign wizard/IPTV calls in
  the original skin were removed.

[COLOR yellow][B]v0.1.19[/B][/COLOR]
* CRITICAL: kill_kodi() is a true hard kill again (os._exit, no Quit).
  The graceful Quit added earlier made Kodi save its in-memory settings
  on shutdown, overwriting the guisettings.xml / Addons33.db that a
  fresh build install had just extracted -- so clean installs rebooted
  into Estuary with everything disabled, while quick updates (whose
  zips carry no guisettings) kept working. This restores the original
  wizard behavior that fresh installs depend on.
* fresh_build_auto_install_if_needed() now skips hydration when
  plugin.video.pov is already on disk (pre-seeded installs, e.g. the
  Windows installer), letting the auto-set-buildname block adopt the
  existing install instead of re-downloading the build over it.

[COLOR yellow][B]v0.1.14[/B][/COLOR]
* Fix the version gate from 0.1.13: _af3_read_addon_version() was matching
  the FIRST version=" in the file, which is the XML declaration
  (<?xml version="1.0"?>), not the <addon> tag. So jurialmunkey 0.2.28 read
  as "1.0", 1.0 >= 0.2.35 passed, and the deps pack was skipped -> the
  TMDbHelper crash persisted after "fixing" it. Now we search from the
  '<addon' tag so the real addon version is read.

[COLOR yellow][B]v0.1.13[/B][/COLOR]
* AF3 deps pack now has a VERSION GATE on jurialmunkey (>= 0.2.35).
  Previously _af3_pack_current() returned True as soon as the addon.xml
  merely existed, so users who switched to AF3 with an old jurialmunkey
  (e.g. 0.2.28) were never upgraded -- but the bundled TMDbHelper 6.15.6
  needs jurialmunkey.ftools (0.2.35+) and crash-looped its service on
  startup, breaking AF3 widgets/ratings. The gate forces a re-extract that
  overwrites the stale module. Version compare is now ">= expected" so a
  newer installed version is never needlessly downgraded.

[COLOR yellow][B]v0.1.11[/B][/COLOR]
* Switch Skin -> Arctic Fuse 3 now actually loads AF3 instead of
  silently falling back to Estuary. Root cause: ensure_arctic_
  fuse_3_installed() extracted the AF3 packs to disk but never
  registered the skin + its 6 dependencies (script.skinvariables,
  script.texturemaker, plugin.video.themoviedb.helper,
  resource.images.weathericons.white, resource.font.robotocjksc,
  resource.images.studios.coloured) in Kodi's Addons DB. From
  Kodi's POV the deps were "not installed", so it refused to load
  AF3 and fell back to skin.estuary -- the "it says switched but I
  get the simple skin" bug. Fix: register + enable every pack's
  addons in the Addons DB after extraction (mirrors quick_update /
  Fresh Install), then UpdateLocalAddons so the dependency graph
  is satisfiable in-session too.
* RETROACTIVE: users who already switched to AF3 with the old
  (DB-less) code are healed automatically -- the "files already on
  disk" fast-path now ALSO re-registers the addons in the DB
  instead of skipping outright. Re-pick AF3 from Switch Skin and
  it will stick after the restart, no re-download needed.
[COLOR yellow][B]v0.1.10[/B][/COLOR]
* quick_update: now passes ignore=True to extract.all so the
  wizard's own files inside the quickfix zip actually get
  written to disk. Without this, every wizard-update shipped
  via quick_update was silently skipped (extract.all has a
  safety guard that excludes anything whose path contains the
  wizard's addon id). End-user symptom: after quick_update +
  restart, Switch Skin kept showing the pre-update list and
  the addon DB version disagreed with the code on disk. PR #161
  Arctic Fuse 3 support was the first user-visible breakage.

[COLOR yellow][B]v0.1.7[/B][/COLOR]
* Python 3.14 compatibility: resources/libs/common/logging.py
  used to do `try: from urllib.request import FancyURLopener
  except ImportError: from urllib import urlencode` -- both
  paths break on Python 3.14 (FancyURLopener was removed,
  `urllib.urlencode` was Py2-only). On Arch Linux + Py3.14 the
  whole wizard refused to start (clear.py -> logging.py import
  chain crashed at module load). Rewrote the failing import to
  a single `from urllib.parse import urlencode`, and replaced
  the `LogURLopener(FancyURLopener)` class + `post_log()` body
  with a `requests.post()` call (requests is already a wizard
  dependency). post_log behaviour is identical; the only
  visible change is that Linux + Py3.14 users can now actually
  launch the wizard.
