v0.2.150
- CRITICAL fix: with "Use community pool" ON, every fresh (non-cached) AI
  translation crashed with "cannot access local variable 'pool'" and fell back
  to the original English ("AI translation failed"). Cause: resolve() uses a
  ThreadPoolExecutor named `pool`, which made Python treat the module-level
  `pool` import as a function-local everywhere in resolve() -- so the pool
  checks earlier in the function referenced it before assignment. Renamed the
  executor to `executor`. Translation works again with the pool enabled.

v0.2.38
- Four targeted improvements to Hebrew gender (זכר/נקבה) accuracy
  in AI translation. Today the gender form is right ~90% of the
  time; the remaining 10% mostly traces to four specific failure
  modes that these changes address. No behavioural change for any
  non-Hebrew code path.

- (a) Cross-chunk context carry-over: we translate in parallel
  chunks (default 250 entries each). Until now, each chunk was
  translated INDEPENDENTLY: the AI saw the cast block but no
  recent conversational thread, so a character finishing a line
  at the end of chunk N would sometimes get a different gender
  the moment they spoke at the start of chunk N+1. We now feed
  the last 5 SOURCE-text lines of chunk N-1 into chunk N's
  prompt as a "PREVIOUS DIALOGUE CONTEXT" section -- AI sees the
  thread, picks up the gender consistently. The previous chunk's
  source is precomputed up-front (deterministic per index) so
  parallel dispatch still works -- no inter-chunk dependency.
  Tunable via the `prev_context_lines` setting (default 5;
  0 disables the feature).

- (b) Speaker-prefix hints: SRT entries that start with an
  ALL-CAPS speaker tag like "MABEL:" or "DR. SMITH:" used to be
  dropped on instruction ("do NOT include speaker prefixes in
  the Hebrew"). The prompt now tells the AI to MATCH the prefix
  name against the cast block, use that person's gender for
  first-person forms in the line, then drop only the prefix
  from the output. Speaker prefixes are common in SDH /
  Telegram SRTs and were previously thrown away even though
  they're the strongest gender signal we ever get.

- (c) Reply heuristic: in two-character dialogues, short
  follow-up lines ("Right.", "No way.", "You sure?", "I told
  you.") are almost always REPLIES from the OTHER speaker.
  New explicit prompt rule: "speakers alternate unless context
  says otherwise; apply alternation before falling back to a
  gender guess." Catches the most common single-line gender
  flip in dialogue.

- (d) Cast list widened from top-12 to top-25 (`tmdb_helper`
  default). Minor characters (waiters, partners, one-scene
  parents) whose gender mattered for the line they spoke were
  often outside the top-12 cut and defaulted to the AI's
  best guess. Existing cached cast lookups with fewer than
  20 entries are treated as cache misses and re-fetched on
  next translation, so users with prior watches get the
  expanded list automatically without manual cache clear.

- Two regressions surfaced on the latest version: (a) iPhone users

- Fix (a): iOS Smart Punctuation replaces ASCII hyphen-minus
  (U+002D) with em-dash (U+2014) for `--` and en-dash (U+2013) for
  a lone `-`. NFKC does NOT collapse these back to ASCII -- they're
  distinct Unicode codepoints. v0.2.34's allow-list regex stripped
  them entirely, so an iPhone-pasted `AIza...XYZ-ABC` arrived at

  rejected the truncated string with 400.

  back to ASCII `-` BEFORE the allow-list runs. Covers 11 variants:
  em-dash, en-dash, minus sign, unicode hyphen, non-breaking
  hyphen, figure dash, horizontal bar, hyphen bullet, full-width
  hyphen-minus, and the small-form variants. Verified against an
  end-to-end PairServer POST with em-dash corruption.

- Fix (b): the old `fentastic_dialog_subtitles_patcher` was
  hard-coded to FENtastic's XML structure. Users on Arctic Zephyr,
  Estuary, Aeon Nox etc. got `'no_file'` from it -- the dialog
  header kept reading `$INFO[Player.Filename]` (the UUID for
  TorBox URLs). The new `skin_dialog_subtitles_patcher`
  auto-detects the active skin via `xbmc.getSkinDir()`, locates
  `DialogSubtitles.xml` in the skin's `xml/`, `720p/`, or `1080i/`
  directory, and uses a regex-based match (case-insensitive on
  the `.Filename` vs `.FileName` capitalisation that varies
  across skins) to swap the label out.
  Self-migrates from the v1 FENtastic-only inject so users
  upgrading don't end up with stale duplicated controls. Verified
  end-to-end against FENtastic, Arctic Zephyr Mod, and Estuary
  XMLs -- all three parse as valid XML after patching, with the
  v2 marker and dual-visibility-gated label pair in place.

- Fix a systemic bug in POV's device-flow auth (Trakt, Real Debrid,
  TorBox, Premiumize, AllDebrid) where the auth dialog would stay
  stuck on screen with the same code/URL even after the user
  authorized on the website.

  Root cause: POV's RepeatTimer.run() in myservices.py polls the
  service's auth endpoint every N seconds. The polling call is
  NOT wrapped in try/except. Any exception from the user-function
  (network timeout, SSLError, malformed JSON, KeyError on
  unexpected response structure) propagates up to threading.Timer
  which doesn't catch it -- the polling thread dies silently and
  the auth detection logic stops running entirely. The dialog
  stays up because nothing was telling it the auth completed.

  This isn't a per-service bug. Any device-flow service POV
  integrates with hits the same fragility -- single transient
  failure on poll #1 = the rest of the user's auth flow is dead.

  Fix is one try/except wrap. Successive polls keep retrying;
  eventually one returns 200 with the token and the auth gets
  captured normally.

  Delivered through a new patcher:
    addons/service.subtitles.kodipovilai/resources/lib/
      pov_repeat_timer_patcher.py
  Wired into service.py's startup hook right after
  _maybe_patch_pov_services() (both modify the same file). Marker
  AI_SUBS_REPEAT_TIMER_PATCH gates idempotency. Defensive: if
  upstream POV restructures RepeatTimer in a way our pattern
  can't match, the patcher skips with a log and POV continues
  working as upstream (auth dialogs go back to being fragile but
  nothing breaks).

  Sandbox tested end-to-end against the current shipped
  myservices.py: first run reports 'patched' and wraps the body
  correctly, re-run reports 'unchanged', py_compile of the
  patched file passes.

      so episodes.py was being needlessly re-copied on every
      Kodi startup (correct behaviour but wasteful logs). Marker
      changed to a substring that all three PR #98 versions share
      and no earlier baseline had:
        kodi_utils.get_setting('trakt_user'
      Also clarified the rewrite path: detection by marker
      presence/absence rather than byte-compare of two files.

- Every patcher now logs explicit INFO/WARNING entries to the
  Kodi log so the next "patcher didn't work" report can be
  diagnosed from the log file alone. Look for
      kodipovilai: <patcher>: <event>
  in xbmc.log -- entries cover not just successes but also

v0.2.15
- Fix the home-screen Favorites tile freezing Kodi. POV ships
  userdata/addon_data/plugin.video.pov/navigator.db with the
  RootList row's Favorites entry pointing at
      mode=navigator.favourites    (UK spelling, with 'u')
  but plugin.video.pov's navigator.py defines the method as
  `def favorites(self):` (US spelling, no 'u'). POV's router

  hits its 5-second script-didn't-finish timeout, the plugin is
  killed, and the user is bounced back to the previous screen.
  Manifests as: click Favorites, Kodi freezes for ~a minute,
  ends up on the home screen.
- Two-pronged fix:
  1. New module resources/lib/pov_navigator_patcher.py opens the
     navigator.db on every Kodi startup and rewrites the bad
     'navigator.favourites' token to 'navigator.favorites' in
     the RootList row. Idempotent (re-run is a no-op),
     defensive (any error path leaves the DB alone), narrow
     (only the one substring is touched). Wired into
     service.py alongside the existing patchers.
  2. The DB shipped in the FENtastic build zip is pre-corrected,
     so future fresh installs don't carry the bug.
- The long-press "Favorites Manager" context-menu and POV's
  own dialogs.favorites_choice were never affected -- they
  call the underlying favorites_cache.Favorites.add() directly
  rather than going through the broken URL. Only the home-
  screen tile was hitting the typo.

  punct to logical start) introduced in v0.2.7 produces the
  correct visual placement. Made it the default; renamed 'auto'
  to 'legacy' and kept it as an option for the rare setup where
  BiDi reorder DOES happen. CACHE_RTL_FIX_VERSION bumped 2 -> 3
  so the migration re-runs over already-cached translations and
  applies the new direction on next Kodi startup. Backwards-
  compat: users who manually picked 'auto' in v0.2.7 settings
  are silently mapped to 'legacy'.

v0.2.7
- User-selectable RTL punctuation fix direction. The previous
  versions assumed Kodi does proper BiDi reorder on subtitle
  lines (logical-end Hebrew punct -> visual-left), so we moved
  the misplaced punct from line start to line end. For some
  Kodi setups -- specific skins, certain Android configurations,
  versions that render source order without RTL reorder -- the
  opposite is true: a logical-START punct ends up at visual-end
  of the reading flow. The user observed this consistently
  across all their subtitles even after the v0.2.5/v0.2.6 fixes.
  New setting `rtl_punct_mode` (Translation -> Behavior) with
  three choices: 'auto' (default, move to end), 'reverse' (move
  to start, for non-BiDi renderers), 'off' (no processing).
  Migration + cache-hit reapply respect the chosen mode.

v0.2.6
- RTL punctuation fix now runs on EVERY cache hit, not just on
  first translation. The v0.2.5 migration (re-running the fix
  over the cached-translation pool) only fires once on Kodi
  startup; users who didn't restart -- or whose cache was
  written between two migration versions -- could still see
  files with the old, broken layout. resolve() now re-applies
  fix_rtl_punctuation in-place on every cache hit (idempotent
  -- if the file is already clean, no write happens), and
  _handle_translate_file applies it once more at the final
  delivery to DarkSubs as belt-and-suspenders. This means the
  user no longer needs to restart Kodi for the fix to roll out
  -- the next time they play a previously-translated subtitle,
  it gets cleaned up on the spot.

v0.2.5
- RTL punctuation fix now handles HTML-wrapped lines. Gemini
  often emits italicised speech as "<i>.לעזאזל</i>" -- the
  previous regex couldn't see past the opening tag and left the
  period at the start. The screenshot the user kept hitting was
  exactly this case. Also covers:
    - period BEFORE the tag: ".<i>לעזאזל</i>" -> "<i>לעזאזל.</i>"
    - dialogue dash + punct + tag: "- <i>.שלום</i>" -> "- <i>שלום.</i>"
    - bold and other tag types (<b>, <font>, ...) -- any tag of
      the shape <name...> at line start.
  Ellipsis still respected (left at start as a continuation
  marker). Existing tests + 16-case unit suite all green.
- CACHE_RTL_FIX_VERSION 1 -> 2 so the migration re-runs over the
  whole cached-translation pool on next startup, applying the
  updated regex to already-saved files that were affected.

v0.2.4
- Toast notifications now scroll right-to-left as Hebrew should.
  Kodi's notification widget infers paragraph direction from the
  first strong character in the message body. Most of our
  notifications start with "AI: ..." (Latin 'A' = strong LTR),
  so the widget set paragraph direction to LTR and scrolled
  left-to-right when the text overflowed. Prepending an invisible
  RIGHT-TO-LEFT MARK (U+200F) in kodi_utils.notify() flips the
  inference to RTL. Single point of fix; applies to every notify
  call site in the addon automatically.
